# TR-005: Implement scheduled job processor with staggered execution

> Task ID: TR-005
> Type: FEATURE
> Priority: P0
> Estimate: 90 minutes
> Assignee: convex-implementer

## CRITICAL CONSTRAINTS

### MUST

- Create scheduled function for processing transcript jobs
- Implement staggered scheduling (1-5 second delays between jobs)
- Follow status machine pattern (pending → transcribing → completed/failed/no_captions)
- Use exponential backoff for retries (2^retryCount seconds)
- Call fetchTranscriptWithFallback service from TR-004
- Update job status with timestamps (startedAt, completedAt)
- Store transcript metadata in videoTranscripts table on success

### NEVER

- Process jobs in parallel (rate limit YouTube API)
- Skip status checks (must guard against terminal states)
- Create infinite loops (max retry count required)
- Throw unhandled errors (catch and mark job as failed)

### STRICTLY

- Process only "pending" status jobs
- Update status to "transcribing" before fetching transcript
- Set startedAt timestamp when status → "transcribing"
- Set completedAt timestamp when status → terminal state
- Use staggered scheduling to prevent API rate limits
- Follow `audio/scheduled.ts` patterns for job processing

## SPECIFICATION

**Objective:** Create scheduled job processor that fetches transcripts with staggered execution and proper error handling.

**Success looks like:** Pending jobs are processed sequentially, transcripts are stored, and job status transitions work correctly.

## ACCEPTANCE CRITERIA

| # | Given | When | Then | Verify |
|---|-------|------|------|--------|
| 1 | Pending job exists | Scheduled function runs | Job status changes to "transcribing" | Job has status="transcribing", startedAt set |
| 2 | Job transcribing | fetchTranscriptWithFallback succeeds | Status changes to "completed" | Job has status="completed", completedAt set, transcriptId populated |
| 3 | Job transcribing | Transcript returned | Stored in videoTranscripts table | Record exists with contentId, storageId, previewText |
| 4 | Job transcribing | fetchTranscriptWithFallback fails (no captions) | Status changes to "no_captions" | Job has status="no_captions", not "failed" |
| 5 | Job transcribing | fetchTranscriptWithFallback errors | Status changes to "failed" | Job has status="failed", errorMessage set |
| 6 | Job failed | Retry attempted | Exponential backoff applied | Retry scheduled after 2^retryCount seconds |
| 7 | Max retries reached | Job processing | Job marked as permanently failed | Status="failed", retryCount > 3, not rescheduled |
| 8 | Multiple pending jobs | Scheduled function runs | Jobs processed with staggered delays | 1-5 second gaps between job starts |

### Interaction Notes

- Staggered scheduling prevents thundering herd on YouTube API
- Status machine guards prevent re-processing completed/failed jobs
- Exponential backoff: 1s, 2s, 4s, 8s for retries 1-4
- Max retry count: 3 (after that, mark as permanently failed)
- Always update job status before external operations (transcript fetch)
- Store transcript metadata only on success, preserve error messages on failure

```typescript
import { internalAction } from "../_generated/server";
import { internal } from "../_generated/api";

export const processSegmentationJobs = internalAction({
  handler: async (ctx) => {
    // Fetch pending jobs
    const jobs = await ctx.runQuery(internal.audio.listPendingJobs, {});

    if (jobs.length === 0) {
      return;
    }

    // Process jobs with staggered scheduling
    const STAGGER_MS = 1000; // 1 second between jobs
    let staggerIndex = 0;

    for (const job of jobs) {
      await ctx.scheduler.runAfter(
        staggerIndex * STAGGER_MS,
        internal.audio.processJob,
        { jobId: job._id }
      );
      staggerIndex++;
    }
  },
});
```

```typescript
export const processJob = internalAction({
  args: { jobId: v.id("assimilateJobs") },
  handler: async (ctx, args) => {
    const job = await ctx.runQuery(internal.assimilate.getJob, { jobId: args.jobId });

    // Status guard: prevent re-processing
    const terminalStatuses = ["completed", "failed", "cancelled"];
    if (terminalStatuses.includes(job.status)) {
      console.log(`Job ${args.jobId} already ${job.status}, skipping`);
      return;
    }

    // Update status to processing
    await ctx.runMutation(internal.assimilate.updateJobStatus, {
      jobId: args.jobId,
      status: "processing",
      startedAt: Date.now(),
    });

    try {
      // Do the work...
      await ctx.runMutation(internal.assimilate.updateJobStatus, {
        jobId: args.jobId,
        status: "completed",
        completedAt: Date.now(),
      });
    } catch (error) {
      await ctx.runMutation(internal.assimilate.updateJobStatus, {
        jobId: args.jobId,
        status: "failed",
        errorMessage: error instanceof Error ? error.message : "Unknown error",
        completedAt: Date.now(),
      });
    }
  },
});
```

Apply these patterns for transcript scheduled processing:

```typescript
// NEW FILE: convex/transcripts/scheduled.ts
import { internalAction } from "../_generated/server";
import { internal } from "../_generated/api";
import { v } from "convex/values";

/**
 * Process pending transcript jobs with staggered scheduling
 * Fetches transcripts via YouTube API (primary) or Jina Reader (fallback)
 */
export const processPendingJobs = internalAction({
  handler: async (ctx) => {
    // Fetch pending jobs, sorted by priority (high first) then createdAt
    const jobs = await ctx.runQuery(internal.transcripts.listPendingJobs, {});

    if (jobs.length === 0) {
      console.log("No pending transcript jobs");
      return;
    }

    console.log(`Processing ${jobs.length} pending transcript jobs`);

    // Stagger jobs to avoid rate limits
    const STAGGER_MS = 2000; // 2 seconds between jobs
    let staggerIndex = 0;

    for (const job of jobs) {
      await ctx.scheduler.runAfter(
        staggerIndex * STAGGER_MS,
        internal.transcripts.processJob,
        { jobId: job._id }
      );
      staggerIndex++;
    }
  },
});

/**
 * Process individual transcript job
 * Handles status transitions, retries, and error recovery
 */
export const processJob = internalAction({
  args: {
    jobId: v.id("transcriptJobs"),
  },
  handler: async (ctx, args) => {
    // Fetch job
    const job = await ctx.runQuery(internal.transcripts.getJob, {
      jobId: args.jobId,
    });

    if (!job) {
      console.error(`Job ${args.jobId} not found`);
      return;
    }

    // Status guard: prevent re-processing of terminal states
    const terminalStatuses = ["completed", "failed", "no_captions"];
    if (terminalStatuses.includes(job.status)) {
      console.log(`Job ${args.jobId} already ${job.status}, skipping`);
      return;
    }

    console.log(`Processing transcript job for ${job.contentId}`);

    // Update status to "transcribing"
    await ctx.runMutation(internal.transcripts.updateJobStatus, {
      jobId: args.jobId,
      status: "transcribing",
      startedAt: Date.now(),
    });

    try {
      // Fetch transcript with fallback (YouTube API → Jina Reader)
      const result = await ctx.runAction(
        internal.transcripts.service.fetchTranscriptWithFallback,
        {
          contentId: job.contentId,
          sourceUrl: job.sourceUrl,
        }
      );

      if (result.success && result.transcript) {
        // Store transcript metadata
        const transcriptId = await ctx.runMutation(
          internal.transcripts.storeTranscript,
          {
            transcript: result.transcript,
          }
        );

        // Update job as completed
        await ctx.runMutation(internal.transcripts.updateJobStatus, {
          jobId: args.jobId,
          status: "completed",
          transcriptId,
          completedAt: Date.now(),
        });

        console.log(`Transcript job completed for ${job.contentId}`);
      } else {
        // No transcript available (not a failure, expected for some videos)
        await ctx.runMutation(internal.transcripts.updateJobStatus, {
          jobId: args.jobId,
          status: "no_captions",
          errorMessage: result.error,
          completedAt: Date.now(),
        });

        console.log(`No captions available for ${job.contentId}`);
      }
    } catch (error) {
      // Handle system errors (API down, network issues, etc.)
      const errorMessage = error instanceof Error ? error.message : "Unknown error";
      console.error(`Transcript job failed for ${job.contentId}:`, errorMessage);

      // Calculate backoff for retry
      const newRetryCount = job.retryCount + 1;
      const MAX_RETRIES = 3;

      if (newRetryCount <= MAX_RETRIES) {
        // Schedule retry with exponential backoff
        const backoffMs = Math.pow(2, job.retryCount) * 1000;

        await ctx.runMutation(internal.transcripts.scheduleRetry, {
          jobId: args.jobId,
          retryCount: newRetryCount,
          errorMessage,
        });

        await ctx.scheduler.runAfter(
          backoffMs,
          internal.transcripts.processJob,
          { jobId: args.jobId }
        );

        console.log(`Scheduled retry ${newRetryCount}/${MAX_RETRIES} for ${job.contentId} after ${backoffMs}ms`);
      } else {
        // Max retries reached, mark as permanently failed
        await ctx.runMutation(internal.transcripts.updateJobStatus, {
          jobId: args.jobId,
          status: "failed",
          errorMessage: `Max retries reached: ${errorMessage}`,
          completedAt: Date.now(),
        });

        console.log(`Job ${args.jobId} permanently failed after ${MAX_RETRIES} retries`);
      }
    }
  },
});
```

### Anti-pattern (DO NOT)

```typescript
// DON'T: Process jobs in parallel
export const processPendingJobs = internalAction({
  handler: async (ctx) => {
    const jobs = await ctx.runQuery(internal.transcripts.listPendingJobs, {});

    // WRONG - Will hit YouTube API rate limits immediately
    await Promise.all(
      jobs.map(job =>

    );
  },
});
```

## NOTES

The staggered scheduling is critical - YouTube API has rate limits and we don't want to hit them. The 2-second delay between jobs means we can process ~30 jobs/minute which is plenty for personal use. The exponential backoff for retries (1s, 2s, 4s, 8s) gives transient failures time to recover without overwhelming the system.
