BACKGROUND
--------------------------------------------------------------------------------

**Problem**: The existing `answer_question` synthesis prompt is generic ("be concise, 2-4 sentences") and gives the LLM no template for recommendation output. Result: even when the right tool fires, the LLM emits prose essays instead of structured numbered lists with names + contacts.

**Why it matters**: This prompt is the deterministic boundary between probabilistic LLM output and the structured data the mobile UI expects. The Zod schema is the hard gate.

**Current state**: `convex/chat/specialistPrompts.ts` has `RESEARCH_SPECIALIST_PROMPT` but no synthesis prompt for recommendations.

**Desired state**: Template-first prompt that produces markdown with name, tagline, rating, location, contact, why-it-fits, source citation — validated against Zod before the mobile card renders.

CODE PATTERN (Reference)

**Source**: `convex/chat/specialistPrompts.ts` existing prompt structure

```ts
import { z } from 'zod';

export const RecommendationSynthesisSchema = z.object({
  items: z.array(z.object({
    name: z.string(),
    description: z.string(),
    whyRecommended: z.string(),
    rating: z.number().optional(),
    location: z.string().optional(),
    pricing: z.string().optional(),
    contact: z.object({
      phone: z.string().optional(),
      url: z.string().optional(),
      email: z.string().optional(),
    }).optional(),
  })),
  sources: z.array(z.object({
    title: z.string(),
    url: z.string(),
    snippet: z.string(),
  })),
});

export const RECOMMENDATION_SYNTHESIS_PROMPT = `
# Role
You synthesize recommendation lists from web sources.

# Output Format (literal markdown template)
1. **{Name}** — {tagline ≤ 80 chars}
   - Rating: {rating if available}
   - Location: {location if available}
   - Contact: {phone/url/email if available}
   - Why it fits: {≤ 1 sentence}
   - Source: [{n}]

# Rules
1. Return EXACTLY {count} entries unless sources don't support that many.
2. Every name must be a real entity from the sources. NEVER GUESS.
3. If a field is missing from the sources, OMIT it — never fabricate.
4. Every entry must cite at least one source by number [n].
5. No preamble. No "I found..." No "Here are...".
6. No conclusion. No "More information". No "Further reading".
7. Tagline ≤ 80 characters.
8. Why-it-fits ≤ 1 sentence.
9. If sources don't yield 3+ providers, return the structured fallback:
   { items: [], sources: [...], note: "Sources didn't yield 3+ named providers for this query." }

# User Criteria
Query: {query}
Count: {count}
Location: {location if any}
Constraints: {constraints if any}

# Sources
{sources}
`;
```

**Anti-pattern**: Do NOT leave optional fields required in Zod; do NOT allow 'Conclusion' sections; do NOT forget the fallback response rule.

**Interaction notes**:
- Template-first output — literal markdown template forces structure
- Zod schema is the post-LLM deterministic gate
