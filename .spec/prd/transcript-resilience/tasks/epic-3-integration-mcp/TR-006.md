# TR-006: Integrate transcript job creation into YouTube subscription flow

> Task ID: TR-006
> Type: FEATURE
> Priority: P0
> Estimate: 60 minutes
> Assignee: convex-implementer

## CRITICAL CONSTRAINTS

### MUST

- Modify `fetchYouTube()` function in `convex/subscriptions/internal.ts`
- Check for existing transcript before creating job (idempotent)
- Create transcript job with priority 5 (medium) after discovering new video
- Link transcript to subscription content via metadata
- Handle transcript job creation failures gracefully (don't block video discovery)
- Log when transcript jobs are created for debugging

### NEVER

- Block video discovery if transcript job creation fails
- Create duplicate transcript jobs for same video
- Modify subscription flow beyond transcript job creation
- Change existing video metadata structure (add fields only)

### STRICTLY

- Use createTranscriptJob mutation from TR-002
- Pass contentId (YouTube video ID) and sourceUrl to job creation
- Set priority=5 for medium priority (creator assimilation can use higher priority)
- Don't await transcript job completion (fire and forget)
- Preserve all existing subscription functionality

## SPECIFICATION

**Objective:** Auto-create transcript jobs when new YouTube videos are discovered via RSS feed.

**Success looks like:** New YouTube videos from subscriptions automatically get transcript jobs created, stored in transcriptJobs table.

## ACCEPTANCE CRITERIA

| # | Given | When | Then | Verify |
|---|-------|------|------|--------|
| 1 | New YouTube video discovered | fetchYouTube() processes video | Transcript job created | Job exists in transcriptJobs with contentId=videoId |
| 2 | Video with existing transcript | fetchYouTube() processes video | No duplicate job created | Idempotency check prevents duplicate |
| 3 | Transcript job created | Job status tracked | Job processed by scheduled function | Status changes from pending → completed/no_captions |
| 4 | Subscription content queried | After job completes | Transcript metadata linked | Content has transcriptId reference |
| 5 | Transcript job creation fails | fetchYouTube() catches error | Video still saved to subscriptionContent | Error logged, video discovery continues |
| 6 | Multiple new videos | fetchYouTube() processes batch | Jobs created for all videos | Each video has corresponding transcript job |

### Code Pattern

Source: `convex/subscriptions/internal.ts` - fetchYouTube() function (existing)

```typescript
async function fetchYouTube(
  source: any,
  filter: RelevancyRule[],
  existingIds: Set<string>
): Promise<Array<{...}>> {
  // ... RSS fetching logic ...

  const newItems = items.filter(item => !existingIds.has(item.id));

  for (const item of newItems) {
    // Save to subscriptionContent
    await ctx.runMutation(internal.subscriptions.saveContent, {
      subscriptionId: source._id,
      contentId: item.id,
      sourceUrl: item.link,
      title: item.title,
      publishedAt: item.pubDate,
      metadata: item.metadata,
    });

    // ADD HERE: Create transcript job
    try {
      await ctx.runMutation(internal.transcripts.createTranscriptJob, {
        contentId: item.id, // YouTube video ID
        sourceUrl: item.link,
        priority: 5, // Medium priority
      });
      console.log(`Created transcript job for YouTube video ${item.id}`);
    } catch (error) {
      // Don't block video discovery if transcript job creation fails
      console.error(`Failed to create transcript job for ${item.id}:`, error);
    }
  }

  return newItems.map(item => ({ ... }));
}
```

## NOTES

The key insight is that transcript job creation should be non-blocking. If the transcript service is down or slow, video discovery should continue. The scheduled job processor (TR-005) will handle transcript fetching asynchronously. This ensures the subscription system remains responsive even if transcript processing is slow.
