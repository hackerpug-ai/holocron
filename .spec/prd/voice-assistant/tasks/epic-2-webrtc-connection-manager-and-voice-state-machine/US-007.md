## CRITICAL CONSTRAINTS

### MUST
- Parse all critical event types: session.created, session.updated, response.output_item.done, response.audio_transcript.done, input_audio_buffer.speech_started, input_audio_buffer.speech_stopped, error
- Emit typed events via callback/EventEmitter pattern for consumers
- Handle response.output_item.done with item.type === 'function_call' separately

### NEVER
- Silently swallow unknown event types — log them at debug level
- Parse events outside the data channel message handler
- Block the event handler with synchronous heavy computation

### STRICTLY
- All event types must be TypeScript typed (discriminated union)
- Filter response.output_item.done by item.type === 'function_call' (event fires for non-function outputs too)

## SPECIFICATION

**Objective:** Create an event handler module that parses data channel messages from OpenAI Realtime and dispatches typed events

**Success looks like:** All OpenAI Realtime data channel events are parsed, typed, and dispatched to registered callbacks

### Interaction Notes
- call_id comes from item.call_id (format: call_XXXX), NOT from item.id (format: item_XXXX)

### Code Pattern
Source: .spec/prd/voice-assistant/technical-requirements/03-function-calling.md:54-70

```typescript
dc.addEventListener("message", (e) => {
  const event = JSON.parse(e.data);

  if (event.type === "response.output_item.done") {
    const { item } = event;
    if (item.type === "function_call") {
      this.callbacks.onFunctionCall?.(item.call_id, item.name, JSON.parse(item.arguments));
    }
  }
});
```

### Anti-pattern (DO NOT)
Do NOT use item.id as call_id — it has format item_XXXX and will cause call_id mismatch errors

## CODING STANDARDS
