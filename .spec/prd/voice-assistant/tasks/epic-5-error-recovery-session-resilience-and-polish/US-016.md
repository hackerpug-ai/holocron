# US-016: Implement service unavailable handling and permission guidance

> Task ID: US-016
> Type: FEATURE
> Priority: P0
> Estimate: 60 minutes
> Assignee: ai-tooling-implementer

## CRITICAL CONSTRAINTS

### MUST
- Detect OpenAI API unavailability (token generation fails or 3 consecutive errors)
- Display 'Voice assistant is currently unavailable' in UI
- Detect microphone permission denial and guide user to platform settings

### NEVER
- Expose API error details to user
- Leave session in broken state after service unavailable
- Skip resource cleanup on any error path

### STRICTLY
- Permission instructions must be platform-specific (iOS vs Android)
- Service unavailable ends session cleanly with no orphaned connections

## SPECIFICATION

**Objective:** Handle service unavailability and permission denial with clear user guidance

**Success looks like:** OpenAI unavailability shows clear message with retry; permission denial shows platform-specific guidance to settings

## ACCEPTANCE CRITERIA

| # | Given | When | Then | Verify |
|---|-------|------|------|--------|
| 1 | OpenAI API is unreachable (createSession fails) | User attempts to start voice session | ERROR state with 'Voice assistant is currently unavailable' message, retry button visible | `pnpm vitest run --reporter=verbose 2>&1 \| grep 'service unavailable'` |
| 2 | Microphone permission not granted | User attempts to start voice session | ERROR state with 'Microphone access needed' message and 'Open Settings' button | `pnpm vitest run --reporter=verbose 2>&1 \| grep 'permission denied'` |
| 3 | User taps retry after service unavailable | Retry is attempted | Fresh connection attempt with new ephemeral token | `pnpm vitest run --reporter=verbose 2>&1 \| grep 'retry fresh'` |
| 4 | Service becomes unavailable mid-session (3 consecutive errors) | Errors accumulate | Session ended cleanly, 'Lost connection to voice service. Please try again.' shown | `pnpm vitest run --reporter=verbose 2>&1 \| grep 'mid-session unavailable'` |
