### Code Pattern
```typescript
// BEFORE — Loads entire table then counts (VERY SLOW)
export const countDocuments = query({
  args: {},
  handler: async (ctx) => {
    const allDocs = await ctx.db.query("documents").collect(); // ❌ Loads everything
    return allDocs.length;
  },
});

export const countDocumentsByCategory = query({
  args: { category: v.string() },
  handler: async (ctx, args) => {
    const allDocs = await ctx.db.query("documents").collect();
    return allDocs.filter(doc => doc.category === args.category).length;
  },
});

// AFTER — Option 1: Use index (if count is small)
export const countDocuments = query({
  args: {},
  handler: async (ctx) => {
    // If you have a counter document:
    const counter = await ctx.db.query("counters")
      .withIndex("by_name", q => q.eq("name", "documents"))
      .first();
    return counter?.count ?? 0;
  },
});

// AFTER — Option 2: Denormalized counter (recommended)
// Add a counter document that's updated on insert/delete:
// In schema:
const counters = defineTable({
  name: v.string(),
  count: v.number(),
}).index("by_name");

// In mutation when creating document:
await ctx.db.insert("counters", {
  name: "documents",
  count: (await ctx.db.query("documents").collect()).length + 1,
});

// In query:
export const countDocuments = query({
  args: {},
  handler: async (ctx) => {
    const counter = await ctx.db.query("counters")
      .withIndex("by_name", q => q.eq("name", "documents"))
      .first();
    return counter?.count ?? 0;
  },
});

// AFTER — Option 3: For filtered counts, use a dedicated counter table
// Create counters per category:
export const countDocumentsByCategory = query({
  args: { category: v.string() },
  handler: async (ctx) => {
    const counter = await ctx.db.query("categoryCounters")
      .withIndex("by_category", q => q.eq("category", args.category))
      .first();
    return counter?.count ?? 0;
  },
});
```

## CODING STANDARDS

- **performance**: Never load entire tables just to count
- **scalability**: Use denormalized counters for frequently accessed counts

## DEPENDENCIES

No dependencies.

## NOTES

- Loading entire tables for counts is O(n) and doesn't scale
- Denormalized counters are O(1) and fast
- Trade-off: counters must be updated on every insert/delete
- For small tables (<100 docs), index-based counting is acceptable
- For large tables, use denormalized counters
- Consider using Convex's built-in count if available (check docs)
- Update counters in mutations, not queries
- Test that counters stay accurate over time
- This is a critical performance fix for large tables
