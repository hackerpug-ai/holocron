## CRITICAL CONSTRAINTS

### MUST
- Pin all GitHub Actions to commit SHAs instead of tags
- Update workflows in `.github/workflows/`
- Verify SHAs are correct

### NEVER
- Use floating tags like `@v1` or `@main`

### STRICTLY
- All action references must use SHAs

## SPECIFICATION

**Objective:** Pin GitHub Actions to specific commit SHAs for security and reproducibility.

**Success looks like:** All workflow files use SHA-pinned action references.

## DESIGN

### References

### Pinning Format
```yaml
# BEFORE — Floating tag (insecure)
- uses: actions/checkout@v3
- uses: actions/setup-node@v3
- uses: dart-lang/setup-dart@v1

# AFTER — SHA-pinned (secure)
- uses: actions/checkout@8f4b7f84864484a7bf31766abe9204da3cbe65b3
- uses: actions/setup-node@5e21ff4d9bc1dd2f3f8d3738a853c3f1fbcd7a13
- uses: dart-lang/setup-dart@e630b99d28a3b71860378cafdc2b3d1213edbc6d
```

### Finding SHAs
```bash
# Get the SHA for an action tag
git ls-remote https://github.com/actions/checkout.git | grep refs/tags/v3
# Or visit: https://github.com/actions/checkout/releases/tag/v3
# Copy the commit SHA shown
```

## CODING STANDARDS

- **security**: Pin to SHAs, not tags
- **reproducibility**: Know exactly which action version runs

## DEPENDENCIES

No dependencies.

## NOTES

- Floating tags can be updated with breaking changes
- SHAs ensure reproducible workflow runs
- Find SHAs from the action's releases page on GitHub
- Document in a comment which tag the SHA corresponds to
- Example: `# actions/checkout@v3 (SHA: 8f4b7f84...)`
- Don't forget third-party actions (EasyEngine, etc.)
- Consider using Dependabot to keep SHAs updated
- Test workflows after pinning to ensure they still work
