## EXAMPLES

**Follow this exact pattern for RLS migration:**

```sql
-- Enable RLS on conversations table
ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;

-- Policy: Authenticated users can only see their own conversations
CREATE POLICY "Users can view own conversations"
ON conversations
FOR SELECT
TO authenticated
USING (auth.uid() = user_id);

-- Policy: Authenticated users can insert their own conversations
CREATE POLICY "Users can insert own conversations"
ON conversations
FOR INSERT
TO authenticated
WITH CHECK (auth.uid() = user_id);

-- Policy: Authenticated users can update their own conversations
CREATE POLICY "Users can update own conversations"
ON conversations
FOR UPDATE
TO authenticated
USING (auth.uid() = user_id)
WITH CHECK (auth.uid() = user_id);

-- Policy: Authenticated users can delete their own conversations
CREATE POLICY "Users can delete own conversations"
ON conversations
FOR DELETE
TO authenticated
USING (auth.uid() = user_id);
```

**Why this pattern:**
- Enables RLS for all operations (SELECT, INSERT, UPDATE, DELETE)
- Uses `auth.uid()` from Supabase Auth for user identification
- Restricts access to authenticated users only
- Isolates data by `user_id` column

**Anti-pattern (DO NOT do this):**

```sql
-- ❌ NO RLS ENABLED - ANYONE CAN ACCESS ANY DATA
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT,
  -- No RLS policies!
);
```
