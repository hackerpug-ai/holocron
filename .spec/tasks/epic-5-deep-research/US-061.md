### MUST
- Add `/cancel` to SUPPORTED_COMMANDS array in slash-commands.ts
- Implement handler that calls `isSessionCancelled()` logic
- Only allow cancellation of user's own sessions (RLS check)

- Return user-friendly confirmation message
- Update session status atomically

## SPECIFICATION

**Objective:** Implement `/cancel` slash command to stop deep research after current iteration

**Success looks like:** User types `/cancel` during active research, system stops after current iteration, user receives confirmation, session status shows 'cancelled'

## DESIGN

### References

### Interaction Notes
- `/cancel` command takes no arguments (uses current conversation's active session)
- If multiple sessions active, cancels the most recent 'running' session
- Status transition: 'running' → 'cancelled' (atomic update)
- Iteration loop checks `isSessionCancelled()` before starting next iteration

### Code Pattern
```typescript
// Source: supabase/functions/chat-send/resume-handler.ts:45-67
const cancelHandler = async (args: string[], userId: string) => {
  const { data: session, error } = await supabase
    .from('deep_research_sessions')
    .select('*')
    .eq('conversation_id', conversationId)
    .eq('status', 'running')
    .single()

  if (error || !session) {
    return { message: 'No active research to cancel' }
  }

  // Verify ownership
  if (session.user_id !== userId) {
    throw new Error('Unauthorized')
  }

  // Atomic status update
  const { error: updateError } = await supabase
    .from('deep_research_sessions')
    .update({ status: 'cancelled' })
    .eq('id', session.id)

  return { message: 'Research will stop after current iteration completes' }
}
```

### Anti-pattern (DO NOT)
```typescript
// ❌ WRONG: Deletes session data

const supabase = createClient(SUPABASE_SERVICE_ROLE_KEY)
```
