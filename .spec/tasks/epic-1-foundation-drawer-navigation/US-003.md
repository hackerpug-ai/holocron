## CRITICAL CONSTRAINTS

### MUST
- Implement all 4 HTTP methods in a single Edge Function file at `supabase/functions/conversations/index.ts`
- Handle GET (list), POST (create), PATCH (rename), DELETE (delete) via HTTP method routing
- GET must return conversations sorted by `updated_at DESC` with a configurable `limit` query parameter (default 50)
- POST must accept an optional `title` field and default to 'New Chat' if omitted
- PATCH must accept `id` and `title` in the request body and update `updated_at` to current timestamp
- DELETE must accept `id` in the request body and cascade-delete all associated chat_messages (via FK ON DELETE CASCADE)
- Return proper HTTP status codes: 200 (success), 201 (created), 400 (bad request), 404 (not found), 405 (method not allowed), 500 (server error)
- Return JSON responses with consistent shape: `{ data: ... }` for success, `{ error: string }` for errors
- Use the Supabase service role key from environment variables for database operations (this is a personal app, no RLS)
- Set CORS headers on all responses to allow requests from the mobile app

### NEVER
- Use RLS (Row Level Security) for this personal app; use the service role key directly
- Import from `lib/supabase.ts` (that is the client-side module); create a server-side Supabase client within the Edge Function
- Return raw database errors to the client; wrap them in user-friendly error messages
- Use `node:` built-in modules; Deno Edge Functions use Deno-native or npm: specifiers
- Hardcode the Supabase URL or keys in the function file; read from `Deno.env.get()`
- Implement authentication or user-scoping; this is a single-user personal app

### STRICTLY
- Follow Supabase Edge Function conventions: `Deno.serve()` handler, `@supabase/supabase-js` client creation per request
- Use TypeScript with strict typing; import the `Conversation`, `ConversationInsert`, `ConversationUpdate` types from the canonical types file or define them inline if import is not possible in Edge Function context
- Validate all required fields before database operations
- The `limit` parameter on GET must be clamped between 1 and 100 to prevent abuse
- The `id` parameter on PATCH and DELETE must be validated as a non-empty string

## SPECIFICATION

**Objective:** Build a single Supabase Edge Function that handles all four conversation CRUD operations (list, create, rename, delete) per the PRD SS11 API specification, enabling the mobile client to manage conversations through the drawer interface.

**Success looks like:** Four HTTP methods on `/functions/v1/conversations` work correctly: GET returns a sorted list of conversations with limit support, POST creates a new conversation and returns it, PATCH renames an existing conversation and updates its timestamp, and DELETE removes a conversation and returns success. All endpoints return well-structured JSON with proper status codes and error handling. The function can be tested locally via `supabase functions serve` and validated with curl commands.

### Code Pattern

```typescript
// supabase/functions/conversations/index.ts
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'

const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type',
  'Access-Control-Allow-Methods': 'GET, POST, PATCH, DELETE, OPTIONS',
}

Deno.serve(async (req: Request) => {
  // Handle CORS preflight
  if (req.method === 'OPTIONS') {
    return new Response(null, { status: 204, headers: corsHeaders })
  }

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!,
  )

  try {
    switch (req.method) {
      case 'GET':
        return await handleList(req, supabase)
      case 'POST':
        return await handleCreate(req, supabase)
      case 'PATCH':
        return await handleRename(req, supabase)
      case 'DELETE':
        return await handleDelete(req, supabase)
      default:
        return jsonResponse({ error: 'Method not allowed' }, 405)
    }
  } catch (error) {
    return jsonResponse({ error: 'Internal server error' }, 500)
  }
})

function jsonResponse(body: unknown, status = 200): Response {
  return new Response(JSON.stringify(body), {
    status,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}
```

### Anti-pattern (DO NOT)

```typescript
// DO NOT import from the app's lib directory
import { supabase } from '../../lib/supabase'  // WRONG: Edge Functions run in Deno, not the app

// DO NOT use node: imports
import { URL } from 'node:url'  // WRONG: Deno has URL globally

// DO NOT return raw Supabase errors
return new Response(JSON.stringify(error), { status: 500 })
// DO: return jsonResponse({ error: 'Failed to list conversations' }, 500)

// DO NOT skip CORS headers
return new Response(JSON.stringify(data))  // WRONG: missing CORS headers

// DO NOT use export default
export default async function handler(req) { ... }  // WRONG: use Deno.serve()

// DO NOT accept unlimited limit values
const limit = parseInt(url.searchParams.get('limit') || '1000')  // WRONG: unbounded
// DO: clamp between 1 and 100
```

## NOTES
- Supabase Edge Functions run in Deno, not Node.js. Use `Deno.serve()`, `Deno.env.get()`, and ESM imports from `https://esm.sh/` or `npm:` specifiers.
- The `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` environment variables are automatically available inside Edge Functions when running via `supabase functions serve` or deployed to Supabase.
- The DELETE operation relies on `ON DELETE CASCADE` from the `chat_messages` foreign key to `conversations.id`. Since `chat_messages` does not exist yet (Epic 2), deleting a conversation in this epic simply removes the conversation row. The cascade will activate once the FK is in place.
- For local testing, start the local Supabase instance with `supabase start`, apply migrations with `supabase db push`, then serve the function with `supabase functions serve conversations --env-file .env.local`.
- The `limit` parameter defaults to 50 per PRD spec but is clamped to a max of 100 to prevent excessive data transfer.
- Consider extracting the CORS headers and Supabase client creation into `_shared/` helper files if the pattern will be reused by future Edge Functions (chat-send, research-start, etc.). This is optional for this task but recommended.
- The PATCH endpoint explicitly sets `updated_at: new Date().toISOString()` rather than relying on a database trigger, since no auto-update trigger was created in US-002.
