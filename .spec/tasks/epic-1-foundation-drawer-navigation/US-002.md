## CRITICAL CONSTRAINTS

### MUST
- Create the `conversations` table exactly matching the schema in PRD SS11 (id UUID PK, title TEXT NOT NULL DEFAULT 'New Chat', last_message_preview TEXT, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ)
- Create the index `idx_conversations_updated ON conversations(updated_at DESC)` for drawer list sorting
- Create the `message_role` ENUM type (`'user'`, `'agent'`, `'system'`) needed by future chat_messages table
- Create the `message_type` ENUM type (`'text'`, `'slash_command'`, `'result_card'`, `'progress'`, `'error'`) needed by future chat_messages table
- Generate TypeScript types that mirror the SQL schema exactly, including nullable fields
- Initialize the Supabase project directory structure (`supabase/`) if it does not already exist
- Use `gen_random_uuid()` for the UUID primary key default (PostgreSQL native, no extension needed on Supabase)
- Use `TIMESTAMPTZ` (not `TIMESTAMP`) for all datetime columns
- Set `DEFAULT NOW()` on both `created_at` and `updated_at` columns

### NEVER
- Create the `chat_messages`, `research_sessions`, `research_iterations`, or `citations` tables in this migration; those belong to later epics
- Use `SERIAL` or `INTEGER` for the conversations primary key; it must be `UUID`
- Use `TIMESTAMP WITHOUT TIME ZONE`; always use `TIMESTAMPTZ`
- Put the migration SQL inside application code; it belongs in `supabase/migrations/`
- Hardcode Supabase project IDs or URLs in migration files
- Modify the existing `lib/supabase.ts` client configuration

### STRICTLY
- Migration filename follows the Supabase convention: `YYYYMMDDHHMMSS_create_conversations.sql` (14-digit timestamp prefix)
- TypeScript types file uses named exports, not default exports
- ENUM types are created with `CREATE TYPE ... AS ENUM` before the table that references them
- The migration must be idempotent-safe: use `IF NOT EXISTS` for type and table creation where supported
- Include a `-- Down migration` comment block at the bottom showing how to reverse the migration

## SPECIFICATION

**Objective:** Create the PostgreSQL migration for the `conversations` table and associated ENUM types per PRD SS11 schema, along with TypeScript type definitions that the mobile client and Edge Functions will use for type-safe database operations.

**Success looks like:** The `supabase/migrations/` directory contains a timestamped SQL file that, when applied via `supabase db push` or `supabase migration up`, creates the `conversations` table, the `idx_conversations_updated` index, and the `message_role` and `message_type` ENUM types. A TypeScript types file at `lib/types/conversations.ts` provides correctly typed interfaces for `Conversation`, `ConversationInsert`, and `ConversationUpdate` that match the SQL schema. The migration applies cleanly to a fresh Supabase instance.

### Code Pattern

Migration SQL:

```sql
-- Migration: Create conversations table and supporting ENUM types
-- PRD Reference: SS11 Data Schema - conversations, message_role, message_type

-- Create ENUM types for future chat_messages table
CREATE TYPE message_role AS ENUM ('user', 'agent', 'system');
CREATE TYPE message_type AS ENUM ('text', 'slash_command', 'result_card', 'progress', 'error');

-- Create conversations table
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL DEFAULT 'New Chat',
  last_message_preview TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Index for drawer list (most recent first)
CREATE INDEX idx_conversations_updated ON conversations(updated_at DESC);
```

TypeScript types:

```typescript
// lib/types/conversations.ts

/** Row type for the conversations table (SELECT result) */
export interface Conversation {
  id: string
  title: string
  last_message_preview: string | null
  created_at: string
  updated_at: string
}

/** Insert type - fields with defaults are optional */
export interface ConversationInsert {
  id?: string
  title?: string
  last_message_preview?: string | null
}

/** Update type - all fields optional */
export interface ConversationUpdate {
  title?: string
  last_message_preview?: string | null
  updated_at?: string
}
```

### Anti-pattern (DO NOT)

```sql
-- DO NOT use SERIAL for the primary key
id SERIAL PRIMARY KEY  -- WRONG: must be UUID

-- DO NOT use TIMESTAMP without timezone
created_at TIMESTAMP DEFAULT NOW()  -- WRONG: use TIMESTAMPTZ

-- DO NOT create chat_messages table here
CREATE TABLE chat_messages (...)  -- WRONG: belongs to Epic 2

-- DO NOT use uuid_generate_v4() without enabling extension
id UUID PRIMARY KEY DEFAULT uuid_generate_v4()  -- RISKY: requires pgcrypto
-- USE: gen_random_uuid() which is built-in to PostgreSQL 13+
```

```typescript
// DO NOT use default exports for type files
export default interface Conversation { ... }  // WRONG

// DO NOT use 'any' types
last_message_preview: any  // WRONG: use string | null

// DO NOT use Date type for timestamp columns
created_at: Date  // WRONG: Supabase returns ISO strings
// USE: created_at: string
```

## NOTES
- The `supabase/` directory does not currently exist in the project root. The agent must run `supabase init` or manually create the directory structure (`supabase/migrations/`, `supabase/config.toml`).
- The ENUM types (`message_role`, `message_type`) are created in this migration even though the `chat_messages` table that uses them comes in a later epic. This avoids migration ordering issues and ensures the types are available when needed.
- The `Conversation` interface in `lib/types/conversations.ts` deliberately uses `string` for timestamp fields (not `Date`) because Supabase returns ISO 8601 strings from its JavaScript client.
- There is already a `Conversation` interface in `screens/DrawerContent.tsx` that is used by the UI components. The new `lib/types/conversations.ts` will become the canonical type; a follow-up task should update `DrawerContent` to import from the canonical location.
- The `updated_at` column does not have an automatic trigger to update on row modification. If auto-update is needed, a separate trigger function can be added in a future migration. For now, Edge Functions will explicitly set `updated_at` on PATCH operations.
