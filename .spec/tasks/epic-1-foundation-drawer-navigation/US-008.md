## CRITICAL CONSTRAINTS

### MUST
- On app launch, automatically navigate to the most recent conversation (by `updated_at` or `created_at`)
- If no conversations exist in the database, create a new "New Chat" conversation and navigate to it
- Use the `useConversations` hook for fetching and creating conversations (no direct Supabase calls in layout files)
- Handle the loading state gracefully -- show a splash/loading screen while determining the initial conversation
- Use Expo Router navigation (`router.replace`) for the initial redirect (replace, not push, to avoid back-stack issues)
- Set the `activeConversationId` in the hook when the initial conversation is determined

### NEVER
- Never show an empty or broken screen while the initial conversation is being determined
- Never navigate before the conversations have finished loading (race condition)
- Never create multiple conversations on first launch due to race conditions or double-renders
- Never skip the loading state -- the user must see feedback during the initial fetch
- Never use `router.navigate` for the initial redirect -- use `router.replace` to prevent back-navigation to a blank screen

### STRICTLY
- The logic must run in the `app/(drawer)/_layout.tsx` file (or a dedicated initializer within the drawer layout)
- The initial navigation must be idempotent -- calling it multiple times should not create duplicate conversations
- If `useConversations` returns conversations, pick `conversations[0]` (already sorted by most recent)
- If `useConversations` returns an empty array, call `createConversation()` once and navigate to the result
- After initial navigation, subsequent app usage follows normal drawer navigation patterns

## SPECIFICATION

**Objective:** Implement the app startup flow that automatically navigates users to their most recent conversation on launch, or creates a new conversation if none exist, ensuring a seamless first-screen experience with no blank or intermediate states.

**Success looks like:** Opening the app shows a brief loading indicator, then the user lands directly on their most recent conversation's chat screen. If they have no conversations (first launch or all deleted), a new "New Chat" conversation is created and they land on it. The drawer sidebar correctly shows this conversation as active.

### Interaction Notes
- The initial navigation runs inside a `useEffect` in the drawer layout after conversations are loaded
- A `hasInitialized` ref prevents duplicate execution on re-renders
- The loading state is shown as the drawer layout's full-screen content (not inside the drawer panel)
- After initial navigation, the layout behaves normally -- the initializer is a one-time setup
- The chat route format is expected to be `/chat/[conversationId]` or `/(drawer)/chat/[conversationId]`

### Code Pattern
```typescript
// app/(drawer)/_layout.tsx
import { useConversations } from '@/hooks/useConversations'
import { useRouter } from 'expo-router'
import { useEffect, useRef } from 'react'
import { ActivityIndicator, View } from 'react-native'
import { Text } from '@/components/ui/text'

export default function DrawerLayout() {
  const router = useRouter()
  const {
    conversations,
    isLoading,
    error,
    createConversation,
    switchConversation,
    refetch,
  } = useConversations()

  const hasInitialized = useRef(false)

  useEffect(() => {
    if (isLoading || hasInitialized.current) return

    const initializeConversation = async () => {
      hasInitialized.current = true

      if (conversations.length > 0) {
        // Navigate to most recent (already sorted)
        const mostRecent = conversations[0]
        switchConversation(mostRecent.id)
        router.replace(`/chat/${mostRecent.id}`)
      } else {
        // No conversations -- create one
        const newId = await createConversation()
        switchConversation(newId)
        router.replace(`/chat/${newId}`)
      }
    }

    initializeConversation()
  }, [isLoading, conversations])

  // Loading state
  if (isLoading) {
    return (
      <View className="bg-background flex-1 items-center justify-center">
        <ActivityIndicator size="large" />
      </View>
    )
  }

  // Error state
  if (error) {
    return (
      <View className="bg-background flex-1 items-center justify-center gap-4 p-6">
        <Text className="text-destructive text-center text-lg">
          Failed to load conversations
        </Text>
        <Text className="text-muted-foreground text-center text-sm">
          {error.message}
        </Text>
        <Button onPress={refetch}>
          <Text>Retry</Text>
        </Button>
      </View>
    )
  }

  // Normal drawer layout
  return (
    <Drawer
      drawerContent={(props) => <CustomDrawerContent {...props} />}
      screenOptions={{ ... }}
    >
      {/* drawer screens */}
    </Drawer>
  )
}
```

### Anti-pattern (DO NOT)
```typescript
// DO NOT: Navigate before loading is complete
useEffect(() => {
  // BAD: conversations might not be loaded yet
  if (conversations.length > 0) {
    router.replace(`/chat/${conversations[0].id}`)
  }
}, []) // Missing isLoading dependency

// DO NOT: Create conversations on every render
useEffect(() => {
  if (conversations.length === 0) {
    createConversation() // BAD: runs on every re-render
  }
}, [conversations]) // Triggers again after creation updates the list

// DO NOT: Use router.navigate for initial redirect
router.navigate(`/chat/${id}`) // BAD: creates back stack entry to blank screen

// DO NOT: Show a blank screen during loading
if (isLoading) return null // BAD: user sees nothing
```

## NOTES
- The `hasInitialized` ref is critical for preventing double-execution. React 18 Strict Mode in development calls effects twice, so without this guard, two conversations could be created on first launch.
- The initial navigation should use `router.replace` (not `router.navigate` or `router.push`) to avoid a blank screen in the back stack. The user should not be able to navigate "back" to the loading screen.
- If the conversations table doesn't exist in Supabase yet, the initial fetch will error. The error state with retry covers this case, but the table must be created (US-003) before this feature works end-to-end.
- Consider whether the "most recent" sort should use `updated_at` (last interaction) or `created_at` (creation time). The PRD says "most recent activity," which implies `updated_at`. Ensure the `useConversations` hook sorts by this field.
- The Storybook toggle in `app/_layout.tsx` (`STORYBOOK_ENABLED`) must remain functional. The drawer layout is a child of the root layout and should not interfere with the Storybook entry point.
