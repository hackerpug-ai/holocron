# Build chat-history Edge Function with cursor-based pagination

> Task ID: US-012
> Type: feature
> Priority: P1
> Appetite Fit: medium (fits)
> Assignee: ai-tooling-implementer
> PRD Version: 3.1.0
> Stability: semi-stable

## CRITICAL CONSTRAINTS

### MUST
- Implement a single Edge Function at `supabase/functions/chat-history/index.ts`
- Accept GET requests with query parameters: `conversation_id` (required), `limit` (optional, default 20), `before` (optional cursor)
- Return messages sorted by `created_at DESC` (newest first) for reverse-chronological display
- Implement cursor-based pagination using the `created_at` timestamp as the cursor
- The `before` parameter is the `created_at` value of the last message in the previous page; return messages older than this
- Return response shape: `{ messages: ChatMessage[], has_more: boolean, next_cursor: string | null }`
- Validate `conversation_id` exists before querying messages
- Return HTTP 200 for success, 400 for missing conversation_id, 404 for conversation not found, 500 for server errors
- Set CORS headers on all responses
- Clamp `limit` between 1 and 100 (default 20)

### NEVER
- Use offset-based pagination (cursor is required for consistent pagination with real-time inserts)
- Return more than 100 messages in a single request
- Include messages from other conversations in the response

### STRICTLY
- The cursor must be the ISO8601 timestamp string from `created_at`
- `has_more` must be `true` if there are messages older than the oldest message returned
- `next_cursor` must be the `created_at` of the oldest message in the current response (for the next page request)
- Messages must include all columns from `chat_messages` table

## SPECIFICATION

**Objective:** Build the `chat-history` Edge Function that returns paginated chat messages for a conversation, enabling infinite scroll in the ChatThread component.

**Success looks like:** GET to `/functions/v1/chat-history?conversation_id=<uuid>` returns the 20 most recent messages. Passing `before=<timestamp>` returns the next page of older messages. `has_more: true` indicates more messages exist; `has_more: false` indicates the user has reached the beginning of the conversation. Edge cases (empty conversation, invalid cursor) are handled gracefully.

### Code Pattern

```typescript
// supabase/functions/chat-history/index.ts
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2'
import { corsHeaders } from '../_shared/cors.ts'

Deno.serve(async (req: Request) => {
  if (req.method === 'OPTIONS') {
    return new Response(null, { status: 204, headers: corsHeaders })
  }

  if (req.method !== 'GET') {
    return jsonResponse({ error: 'Method not allowed' }, 405)
  }

  const url = new URL(req.url)
  const conversationId = url.searchParams.get('conversation_id')
  const limitParam = url.searchParams.get('limit')
  const before = url.searchParams.get('before')

  // Validate required params
  if (!conversationId) {
    return jsonResponse({ error: 'conversation_id is required' }, 400)
  }

  // Clamp limit between 1 and 100, default 20
  let limit = parseInt(limitParam || '20', 10)
  if (isNaN(limit) || limit < 1) limit = 20
  if (limit > 100) limit = 100

  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!,
  )

  try {
    // Verify conversation exists
    const { data: conversation, error: convError } = await supabase
      .from('conversations')
      .select('id')
      .eq('id', conversationId)
      .single()

    if (convError || !conversation) {
      return jsonResponse({ error: 'Conversation not found' }, 404)
    }

    // Build query for messages
    let query = supabase
      .from('chat_messages')
      .select('*')
      .eq('conversation_id', conversationId)
      .order('created_at', { ascending: false })
      .limit(limit + 1)  // Fetch one extra to determine has_more

    // Apply cursor filter if provided
    if (before) {
      query = query.lt('created_at', before)
    }

    const { data: messages, error: msgError } = await query

    if (msgError) throw msgError

    // Determine pagination state
    const hasMore = messages && messages.length > limit
    const resultMessages = hasMore ? messages.slice(0, limit) : (messages || [])
    const nextCursor = resultMessages.length > 0
      ? resultMessages[resultMessages.length - 1].created_at
      : null

    return jsonResponse({
      messages: resultMessages,
      has_more: hasMore,
      next_cursor: hasMore ? nextCursor : null,
    })

  } catch (error) {
    console.error('chat-history error:', error)
    return jsonResponse({ error: 'Internal server error' }, 500)
  }
})

function jsonResponse(body: unknown, status = 200): Response {
  return new Response(JSON.stringify(body), {
    status,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}
```

### Anti-pattern (DO NOT)

```typescript
// DO NOT use offset pagination
.range(offset, offset + limit)  // WRONG: cursor-based is required

// DO NOT forget to check for has_more
return { messages }  // WRONG: must include has_more and next_cursor

// DO NOT return messages in wrong order
.order('created_at', { ascending: true })  // WRONG: should be DESC for newest-first

// DO NOT include messages from other conversations
.select('*')  // Missing .eq('conversation_id', conversationId)
```

## NOTES
- The "fetch limit+1" pattern is a common way to determine `has_more` without a separate COUNT query.
- `next_cursor` is the `created_at` of the oldest message returned, used as `before` in the next request.
- Messages are returned newest-first (DESC) because the UI displays newest at the bottom with infinite scroll loading older messages on scroll-up.
- For real-time new messages, the client uses Supabase Realtime subscriptions (US-018), not polling this endpoint.
- The index `idx_chat_messages_conversation` on `(conversation_id, created_at DESC)` ensures this query is efficient.
